1. family_hx_crc FIX IS IN THE CODE BUT NOT IN THE CURRENT report.md/model
===============================================================================
Status: FIXED IN CODE (commit 677b220), NOT YET RE-RUN.

Root cause (confirmed from feature_audit.md against real Minerva data):
Questionnaire.FAM_HX_COLON_CANCER stores WHO has the family history
("Father", "Mother's Parents", "You", "Siblings", ...) -- the SAME "who has
it" relation encoding as the PERS_HX_* columns, NOT a Yes/No flag. The old
config matched against positive_values ["yes","y","1","true"], which never
appeared in the real data -- so family_hx_crc was silently ALWAYS 0 in every
run to date, including the one whose report.md/PNGs went into the
presentation.

Fix applied: features.py now matches whole-word "You" (regex \bYou\b) on
FAM_HX_COLON_CANCER, exactly like PERS_HX_IBD/DIABETES/OBESITY/HTN.

TO PICK THIS UP: rerun features.py -> build_dataset.py -> train.py ->
evaluate.py -> external_validate.py -> report.py (the lsf/rerun_fast.bsub
path is fine -- this fix doesn't touch preprocess/phenotype/match).
Family history of CRC is a clinically meaningful risk factor; expect it to
show up with real (small but nonzero) importance once this reruns, and the
carrier-enrichment / SHAP story to shift slightly.
